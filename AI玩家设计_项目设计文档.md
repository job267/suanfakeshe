# AI玩家设计 — 项目设计文档

## 一、项目概述

本项目的AI玩家需要从迷宫起点出发，在有限视野下穿越迷宫、拾取金币、避开陷阱、击败守卫BOSS，最终抵达终点。核心算法为**贪心策略**（实时资源拾取），辅助算法包括A*路径规划、BFS最短路径、分支限界BOSS战，以及可选增强的Q-Learning强化学习对比。

---

## 二、课题目标与任务约束

### 2.1 来自任务书的要求

| 任务项 | 输入 | 输出 | 核心算法 |
|--------|------|------|----------|
| 实时资源拾取 | AI当前位置、视野内金币和陷阱分布 | 拾取资源价值、拾取路径可视化 | 贪心 |
| 完整迷宫探险 | 完整迷宫矩阵 | 过程可视化、剩余资源/步数比值 | 贪心+A*/BFS组合 |

### 2.2 评分标准

- **阶段测试**：贪心拾取的平均资源价值（多个3×3局部用例） + 完整探险在不同迷宫上的剩余资源/步数比值
- **交叉测试**：所有迷宫设计组的"最佳迷宫"组成基准测试集，AI玩家需全部通过，分数由全部迷宫的测评结果决定
- **报告**：算法伪代码或流程图、时空复杂度分析、实验数据图表、创新性分析

### 2.3 来自迷宫设计方的约束

迷宫设计组提供标准化的 JSON 迷宫文件，包含迷宫矩阵和 BOSS/技能参数。**外部交换格式**与**内部数据模型**分离——外部使用字符符号便于人类阅读和编辑，内部转为整数编码便于算法处理。

#### 2.3.1 外部交换格式（JSON，来自迷宫设计方）

```json
{
    "maze": [["#","S",...,"E"], ...],   // 字符矩阵
    "B": [k0, k1, ..., kn-1],           // BOSS群血量数组
    "PlayerSkills": [[a0,b0], ...],      // 玩家技能 [伤害, 冷却]
    "minRouds": 20,                      // 击败BOSS限定回合数
    "CoinConsumption": 5                 // 复活消耗金币数
}
```

迷宫矩阵为二维字符数组，元素定义如下：

| 符号 | 含义 | 说明 |
|------|------|------|
| `#` | 墙壁 | 不可通行 |
| ` ` (空格) | 普通通路 | 可通行 |
| `S` | 起点 | AI 初始位置 |
| `E` | 终点 | 目标位置 |
| `G` | 金币 | 拾取后获得 **+50** 资源，格子变为通路 |
| `T` | 陷阱 | 触发后扣除 **-30** 资源，仅触发一次，之后变为通路 |
| `B` | BOSS守卫 | 触发 BOSS 战，击败后变为通路 |

除 `#` 外，其他字符都视为可通行格子。两个可通行格子如果上下左右相邻，则它们之间存在一条边。

**BOSS 配置说明：**
- `B` 字段为数组 `[k0, k1, ..., kn-1]`，包含 n 个 BOSS 的血量
- 迷宫中 `B` 标记**唯一**，代表一处 BOSS 群，击败顺序固定（先击败 HP=k0 的 BOSS，再击败 HP=k1 的，依次类推）
- `minRouds`：每轮击败 BOSS 的限定回合数（全局共享）
- `CoinConsumption`：未在限定回合内击败时需要消耗的金币数以复活重试（全局共享）

**玩家技能说明：**
- `PlayerSkills` 为 `[[伤害, 冷却], ...]` 的二维数组，如 `[[8,4],[2,0],[4,2],[6,3]]`
- 至少包含一个冷却为 0 的技能（无冷却普攻）
- 技能不携带名称，内部自动生成标识（如"技能-0""技能-1"）

#### 2.3.2 内部数据模型（整数编码）

外部字符在加载时统一转换为内部整数编码，算法模块仅依赖整数模型：

| 外部符号 | 内部编码 | 常量名 |
|----------|:--------:|--------|
| `#` | 0 | `WALL` |
| ` ` | 1 | `ROAD` |
| `S` | 2 | `START` |
| `E` | 3 | `END` |
| `G` | 4 | `COIN` |
| `T` | 5 | `TRAP` |
| `B` | 6 | `BOSS` |

AI玩家视野**受限于周围3×3区域**，无法看到全局地图，但可以记忆已访问区域的信息。

## 四、基础数据层设计

### 4.1 Maze 数据结构

内部统一使用 `Maze` 对象作为数据载体，与外部 JSON 文件格式解耦。外部字符符号在加载阶段转换为内部整数编码（0~6），算法模块仅操作整数矩阵。

**核心属性：**

| 属性 | 类型 | 说明 |
|------|------|------|
| `name` | 字符串 | 迷宫名称（默认取文件名） |
| `matrix` | 二维整数数组 | 每个格子的类型（0~6 整数编码） |
| `bosses` | `dict[Pos, Boss]` | BOSS 位置到 Boss 对象的映射（支持多 BOSS） |
| `min_rounds` | 整数 | 每轮击败 BOSS 的限定回合数（全局共享） |
| `coin_consumption` | 整数 | BOSS 战复活消耗金币数（全局共享） |

**推导属性：**

| 属性 | 类型 | 说明 |
|------|------|------|
| `height` / `width` | 整数 | 迷宫行列数 |
| `start` | 坐标对 | 起点位置（`START=2` 所在格子） |
| `end` | 坐标对 | 终点位置（`END=3` 所在格子） |
| `total_coins` | 整数 | 迷宫中金币总数 |

**Boss 数据模型：**

```python
@dataclass(frozen=True)
class Boss:
    boss_id: str        # 自动生成，如 "BOSS-1"
    position: Pos       # BOSS 所在坐标
    hp: int             # 血量（来自外部 B 数组对应元素）
    max_rounds: int     # 限定回合数（取自全局 minRouds）
    revive_cost: int    # 复活代价（取自全局 CoinConsumption）
```

**Skill 数据模型：**

外部输入为 `[damage, cooldown]` 元组，内部转换为 Skill 对象：

```python
@dataclass(frozen=True)
class Skill:
    name: str       # 自动生成，如 "技能-0"
    damage: int     # 技能伤害值
    cooldown: int   # 冷却回合数（0 = 无冷却）
```

**DEFAULT_SKILLS — 技能回退机制：**

当外部 JSON 未提供 `PlayerSkills` 字段时（如内部测试用例或旧格式文件），使用预定义的默认技能列表作为回退：

```python
DEFAULT_SKILLS = [
    Skill("普通攻击", damage=5, cooldown=0),
    Skill("重击", damage=12, cooldown=2),
]
```

此回退仅用于兼容场景；正式的外部迷宫文件必须提供 `PlayerSkills` 字段，校验器会对此进行检查（见 4.3）。

**基本操作：**

- 获取指定位置的可通行相邻格子（上下左右，排除墙壁）
- 获取指定位置为中心、指定半径内的视野范围
- 判断某个位置是否在迷宫范围内
- 深拷贝矩阵（用于算法对比时的隔离运行）
- 获取指定格子类型（支持传入替代矩阵用于模拟）

### 4.2 迷宫文件加载 — 策略模式 + 符号转换

**当前状态：与迷宫设计组的数据交换格式已确定。** 采用策略模式，在外部 JSON 文件和内部 Maze 对象之间插入解析层，外部字符符号在此层统一转换为内部整数编码。

**架构：**

```
外部 JSON 文件 (字符符号: # S E G T B)
       │
       ▼
  MazeParser 抽象接口
       ├─ JsonMazeParser      ← 当前实现：解析 JSON 字符矩阵
       └─ (新格式只需新增子类)
       │
       ├─ 字符→整数映射转换
       │   '#': 0, ' ': 1, 'S': 2, 'E': 3, 'G': 4, 'T': 5, 'B': 6
       │
       ▼
  Maze 对象 (内部统一整数矩阵)
       │
       ▼
  APIValidator 校验
```

**JsonMazeParser 解析流程：**

1. 读取 JSON 文件，提取 `maze`（字符矩阵）、`B`（血量数组）、`PlayerSkills`、`minRouds`、`CoinConsumption`
2. 将字符矩阵逐格转换为整数矩阵（按 2.3.2 映射表）
3. 扫描矩阵中所有 `B` 位置，按从上到下、从左到右的顺序与 `B` 数组中的血量值一一对应，构造 Boss 对象
4. 将 `PlayerSkills` 的 `[dmg, cd]` 元组转为 Skill 对象（自动命名为"技能-0""技能-1"...）
5. 将全局 `minRouds` 和 `CoinConsumption` 分别赋予每个 Boss 的 `max_rounds` 和 `revive_cost`
6. 返回完整的 Maze 对象

**MazeLoader**：统一加载入口，内部维护一个 Parser 注册表。根据文件后缀自动分发到对应的 Parser。外部调用者只需传文件路径，不感知具体格式。

**格式变更的影响范围：** 若迷宫设计方调整字段名或新增字段，仅需修改 JsonMazeParser 的转换逻辑，Maze 模型、算法模块、PyQt 窗口均不受影响。

**内置预设迷宫（`sample_mazes.py`）：** 除从外部 JSON 文件加载外，项目还提供了 4 个硬编码的预设迷宫布局，用于开发期快速验证和演示。这些迷宫通过 `SYMBOLS` 字典定义字符→整数映射，在代码中直接以字符串形式编写布局，通过 `MazeLoader.from_layout()` 加载。预设迷宫包括："资源回廊"（密集金币）、"陷阱绕行"（金币与陷阱交替）、"BOSS关卡"（必经 BOSS）、"综合挑战15x15"（全要素综合）。当需要新增测试迷宫时，推荐直接编写 JSON 文件放入 `examples/` 目录，而非在 `sample_mazes.py` 中硬编码。

### 4.3 APIValidator — 输入输出校验

在 Maze 对象构造后、算法运行前，自动校验：

| 校验项 | 规则 |
|--------|------|
| 矩阵完整性 | 矩阵非空，所有行长度一致 |
| 必要元素 | 必须有且仅有一个起点(2)和一个终点(3) |
| 连通性 | 起点到终点之间存在通路 |
| 字符合法性 | 外部矩阵仅包含 `#` ` ` `S` `E` `G` `T` `B` 七种字符 |
| BOSS参数 | BOSS 位置在矩阵中标记为 6；血量来自 B 数组且为正整数；`minRouds` > 0；`CoinConsumption` >= 0 |
| BOSS数量一致性 | 矩阵中 `B` 的数量 = B 数组长度 |
| 技能合法性 | `PlayerSkills` 非空，至少有一个 `cooldown=0` 的技能 |

## 五、算法核心层设计

### 5.1 贪心资源拾取（任务① 必做）

**算法思路：** AI在每一步移动前扫描3×3视野内的所有资源（金币和陷阱），对每个资源计算"性价比"——资源价值除以曼哈顿距离。按性价比降序排列，选择性价比最高且可到达的资源作为目标，用BFS规划到达该资源的最短路径，移动一步。重复此过程。

**关键设计决策：**

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 性价比公式 | `价值 / (曼哈顿距离 + 1)` | 距离为0时除1防零，距离越远性价比越低 |
| 陷阱处理 | 价值为-30，性价比为负时直接排除 | 负收益目标不应被贪心选中 |
| 视野外资源 | 不可见，不参与决策 | 模拟真实有限视野约束 |
| 视野内无资源时 | 向终点方向移动一步 | 保证不断向终点推进 |
| 已访问记忆 | 已捡金币和已触发陷阱标记后不再重复计算 | 利用记忆弥补视野限制 |

**输入：** Maze对象、当前AI位置、已收集金币集合、已触发陷阱集合、视野半径（默认1）

**输出：** 决策结果包含两部分——

| 字段 | 类型 | 说明 |
|------|------|------|
| `next_pos` | 坐标 | 下一步移动目标位置（核心输出） |
| `coins_in_view` | 列表 | 视野内每个金币的 `(坐标, 距离, 性价比)` |
| `traps_in_view` | 列表 | 视野内每个陷阱的 `(坐标, 距离)` |
| `selected_target` | 字符串或空 | 选中目标的描述，如"金币(6,8) 性价比=25.0"，无资源时为空 |

核心算法逻辑（扫描视野 → 算性价比 → 排序选最优）不变，仅在原有输出基础上附加推理详情，供前端展示决策过程。

**复杂度：** 每步决策 O(k + V)，其中 k 为视野内格子数（3×3=9），V 为到选中目标的BFS代价（worst-case O(n²)）

### 5.2 路径规划（支撑模块）

为贪心拾取和完整探险提供路径查询能力。

| 算法 | 适用场景 | 特点 |
|------|----------|------|
| BFS | 到目标资源的最短路径 | 无权图最优，O(n²) |
| A* | 到终点的方向性导航 | 曼哈顿距离启发式，实际运行更快 |
| Dijkstra | 带陷阱权重的路径规划（可选） | 将陷阱代价纳入路径选择 |

**缓存策略：** A* 路径不缓存，每步重新计算。理由：15×15 迷宫上 A* 耗时 < 1ms，每步重算足够；且迷宫状态可能因 BOSS 击败后通路变化需要更新路径。对于连续步中向同一目标移动的场景，实际开销可忽略。

### 5.3 BOSS战 — 分支限界（辅助必做）

**场景：** AI移动至BOSS格子时触发战斗。BOSS战在一个独立的回合制框架中进行，AI玩家技能来自外部 `PlayerSkills` 数组（格式为 `[[伤害, 冷却], ...]`，如 `[[5,0],[10,2]]` 表示技能-0 伤害5无冷却、技能-1 伤害10冷却2回合）。外部未提供技能时，使用内置默认技能回退。

**模式分离：求解模式 与 执行模式**

前端需要在 BOSS 战时实时显示血条变化、回合推进和技能释放，而算法对比平台需要快速拿到最优解。两者需求不同，因此拆为两个函数：

| 模式 | 函数 | 职责 | 调用方 |
|------|------|------|--------|
| 求解模式 | `boss_battle_optimal(boss, skills)` | 用分支限界搜索最少回合数和最优技能序列 | 算法对比平台、理论分析 |
| 执行模式 | `boss_battle_execute(boss, skills, sequence, progress_callback)` | 按给定技能序列逐步执行，每回合回调 `(round, hp_remaining, skill_used)` 并检查回调返回值以响应终止请求 | PyQt 实时探险窗口 |

**求解模式算法思路：** 将BOSS战建模为状态空间搜索问题。状态由（BOSS剩余血量、各技能冷却剩余回合列表——列表长度动态匹配技能数量、已用回合数）组成。每个状态的分支是所有当前可释放的技能。使用分支限界法搜索最少回合数的技能释放序列，以当前最优回合数作为剪枝界。

**执行模式终止响应：** `progress_callback` 每回合返回布尔值——返回 `True` 继续执行，返回 `False` 表示用户请求终止。执行模式收到 `False` 后立即中断、标记"用户终止"并返回。避免用户点"终止"后需等待战斗结束的体验问题。

**技能数量适配：** 分支限界的状态空间中，技能冷却列表为动态长度 `[cd_1, cd_2, ..., cd_k]`，其中 k 等于实际技能数量（迷宫设计方可能提供 1~4 个技能），不假设固定数量。

**关键设计：**

| 设计项 | 说明 |
|--------|------|
| 状态表示 | (boss_hp, cooldown_1, cooldown_2, ..., round) |
| 分支策略 | 每个可用技能对应一个分支 |
| 剪枝界 | 当前已找到的最少回合数（上界），超过即剪枝 |
| 复活机制 | 若回合数超限，消耗金币复活重试；金币耗尽则GAME OVER |
| 最优序列 | 分支限界直接输出最少回合数和对应技能序列 |

**执行模式流程：** 先调用求解模式拿到最优序列，然后逐回合执行。每回合调用 `progress_callback(round, hp_remaining, skill_used)`，前端收到回调后更新血条进度条和技能日志。

**输入：** BOSS参数（血量来自外部 `B` 数组、限定回合数来自 `minRouds`、复活金币数来自 `CoinConsumption`）、AI技能列表（来自外部 `PlayerSkills` 数组，内部自动命名为"技能-0""技能-1"等）

**输出：** 最少回合数、最优技能序列（技能名称列表）、实际战斗胜负、复活次数、血量历史（用于绘图）

### 5.4 完整迷宫探险（任务② 必做）

**两种运行模式：**

| 模式 | 调用方式 | 使用场景 |
|------|----------|----------|
| 批量模式 | `exploration.run_full(maze) → AlgorithmResult` | Console调试、算法对比平台、测试 |
| 步进模式 | QThread worker 中逐步骤调用各子模块 | PyQt 实时探险窗口 |

**批量模式流程：** 与原始设计一致，输入 Maze，内部循环一步到底，返回完整 `AlgorithmResult`。`exploration.py` 保留此模式，不改接口。

**步进模式流程：** 探险主循环位于 `ExplorationWorker.run()` 中（QThread）。worker 在构造时接收一个 **Strategy 对象**，主循环通过策略接口调用决策逻辑，不硬编码具体算法：

```
ExplorationWorker(maze, strategy)     ← strategy 是接口对象
       │
       ▼
  worker.run() 主循环:
       │
       ├─ 1. strategy.decide(maze, pos, collected, triggered) → Decision
       ├─ 2. 移动至 Decision.next_pos
       ├─ 3. 检查格子类型，更新资源和状态
       ├─ 4. 若遇 BOSS：求解 → 逐步执行
       ├─ 5. 组装 StepSnapshot → emit 信号 → msleep(delay)
       └─ 6. 若步数超限 → 降级策略
```

**Strategy 接口与实现：**

```
ExplorationStrategy (抽象接口)
  │
  ├─ GreedyStrategy      → 调 greedy_pick.decision()，寻路用 BFS
  ├─ AstarStrategy       → 调 A* 向终点方向，沿途捡视野内金币
  ├─ BfsStrategy         → 纯 BFS 最短路径，不主动收集资源
  └─ AdaptiveStrategy    → 自适应权重贪心（后续新增，写一个类即可）
```

策略类统一放在 `ai_player/strategies.py` 中。每个策略实现两个方法：
- `decide(maze, pos, collected, triggered) → Decision` — 核心决策
- `name → str` — 标识名称

**新增策略的步骤：** 写一个 Strategy 子类 → 在对比平台注册 → 在 PyQt 算法选择下拉框中出现。Worker 和 exploration 代码零改动。

这种编排方式下，`exploration.py` 的批量模式保持不变，QThread worker 通过策略接口编排子模块调用。

**流程：**

```
初始化(起点, 资源=0, 步数=0)
     │
     ▼
┌──────────────────────────┐
│  主循环 (直到到达终点)    │
│                          │
│  1. 贪心扫描3×3视野       │
│     ├─ 有资源 → 选最优 → 移动一步                │
│     └─ 无资源 → A*向终点移动一步                 │
│                          │
│  2. 移动后检查格子类型     │
│     ├─ 金币 → 拾取(资源+50)                       │
│     ├─ 陷阱 → 触发(资源-30, 标记已触发)           │
│     └─ BOSS → 触发BOSS战(求解→逐步执行→回调UI)    │
│                          │
│  3. 组装StepSnapshot → emit信号 → msleep(delay)  │
│                          │
│  4. 若步数超限 → 降级策略  │
└──────────────────────────┘
     │
     ▼
输出结果 (路径, 剩余资源, 步数, 效率比值)
```

**降级策略（三层防御）：**

| 层级 | 触发条件 | 策略 | 目标 |
|------|----------|------|------|
| Layer 1 | 默认 | 自适应贪心 + 记忆地图 + A*导航 | 追求高分 |
| Layer 2 | 连续N步资源无增长 | 纯A*最短路径到终点，沿途仅捡路径上的金币（不绕路） | 保证有分数 |
| Layer 3 | BOSS战失败且金币不足，或遭遇无法绕过的BOSS | 沿BFS路径直行，遇BOSS仅用无冷却普攻（不追求最优序列），用完限定回合后若失败则复活重试，金币彻底耗尽则标记为"不可通过"并报告 | 保证不进入死循环，产出明确失败结果 |

**BOSS战前置资源检查：** 在进入 BOSS 格之前，先判断当前资源是否 ≥ 复活代价。若资源不足且该 BOSS 在必经之路上（无可绕行路径），则直接降级到对应层级，避免先打再失败浪费执行。

**输出指标：** 完整移动路径、最终剩余资源价值、总步数（含迷宫步数与BOSS回合数）、效率比值（资源/步数）、BOSS战记录、过程可视化数据。步数指标内部区分为 `maze_steps`（移动格子数）和 `boss_rounds`（BOSS战回合数），便于分析"探索效率"与"战斗效率"各自的贡献。

### 5.5 Q-Learning 增强（可选创新模块）

**定位：** 非必做，作为贪心策略的对比参照系，为报告的"算法创新性"提供素材。

**设计思路：** 使用表格型 Q-Learning（零深度学习依赖），将迷宫环境建模为离散状态空间，AI的四个移动方向（上下左右）为动作空间。通过奖励函数引导学习——金币+50、陷阱-30、到达终点+200、撞墙-5。使用 ε-greedy 策略训练，ε 从1.0线性衰减到0.01。

**状态离散化策略：** 不对原始 15×15=225 个位置直接编码，而是用相对位置（到终点的偏移量）+ 四个方向上的最近资源方向/距离（分桶为相邻/中距/远距）。预估 Q-table 大小在 5000~20000 个状态之间，内存可控。

**训练与泛化：** 在5×5小迷宫上训练5000轮，在15×15迷宫上测试泛化能力。与贪心策略在相同的剩余资源/步数指标上比较。

### 5.6 后端与前端的数据契约

为统一后端算法模块与前端展示（Console/PyQt/Matplotlib）之间的数据交换，定义以下数据类，存放在 `types.py` 中。这些是纯数据结构，不包含业务逻辑。

| 数据类 | 用途 | 传递方向 | 包含字段 |
|--------|------|----------|----------|
| `StepSnapshot` | 每一步的状态快照 | 算法 → UI 信号 | 步数、位置、资源值、已收集/已触发集合、视野内金币列表（含性价比）、视野内陷阱列表、选中目标描述、**BOSS状态列表**（每个BOSS一条：`{id, position, status, hp}`）、视野半径 |
| `BossResult` | 单次BOSS战的结果 | BOSS模块 → UI 信号 / 对比引擎 | BOSS标识、胜负、回合数、复活次数、血量历史（每回合）、技能释放序列 |
| `AlgorithmResult` | 单个算法完整运行结果 | 各算法 → 对比引擎 / 测试框架 | 算法名、**迷宫步数(maze_steps)、BOSS回合数(boss_rounds)、总步数**、净资源、金币收集数/总数、陷阱触发数、BOSS结果列表、完整路径、执行耗时、是否到达终点、效率比值 |
| `CompareResult` | 多算法对比汇总 | 对比引擎 → 前端展示 | 迷宫名称、各算法结果列表、各指标最优算法名 |

**关于 StepSnapshot 数据量：** 每帧携带完整的 `collected_positions` 和 `triggered_positions` 集合。15×15 迷宫最多 225 个坐标，每条数据约 2KB，200 步累计约 400KB——在信号传输中完全可控。采用完整快照而非增量（delta）设计，确保前端可从任意帧独立还原全部状态，便于进度条拖动到任意位置跳帧。

**BOSS状态从单值改为列表：** 一个迷宫可能有 2~3 个 BOSS，各自独立存在。`StepSnapshot.boss_status` 从单个字符串改为列表 `[{boss_id, position, status, hp_remaining}, ...]`，支持"BOSS#1已击败，BOSS#2待挑战"这种复合状态。

**数据流向：**

```
greedy_pick ──→ StepSnapshot ──→ ExplorationWorker ──信号──→ MazeWindow
boss_battle ──→ BossResult    ──→ ExplorationWorker ──信号──→ MazeWindow

exploration.run_full() ──→ AlgorithmResult ──→ AlgorithmComparator ──→ CompareResult
rl_agent.run()         ──→ AlgorithmResult ──┘
```

## 七、算法对比平台设计

### 7.1 功能概述

在同一迷宫上运行多种策略，结果并排对比。位于 PyQt 窗口右侧的"算法对比"标签页中。直接为报告的"实验对比分析"章节提供数据和图表。

### 7.2 对比维度

| 指标 | 方向 | 含义 |
|------|:--:|------|
| 迷宫步数 | ↓ | 在迷宫中移动的总格子数（不含BOSS回合） |
| BOSS回合数 | ↓ | BOSS战中消耗的总回合数 |
| 总步数 | ↓ | 迷宫步数 + BOSS回合数 |
| 净资源价值 | ↑ | 金币收入 - 陷阱损失 - 复活消耗 |
| 效率 | ↑ | 资源价值 / 总步数（核心指标） |
| 探索效率 | ↑ | 净资源 / 迷宫步数（排除BOSS影响，纯衡量探索策略） |
| 金币收集率 | ↑ | 实际收集金币数 / 迷宫金币总数 |
| 陷阱触发数 | ↓ | 触发陷阱总次数 |
| BOSS战胜率 | ↑ | 首次战胜 / 总尝试次数 |
| 执行时间 | ↓ | 算法 Wall-Clock 时间 |

### 7.3 架构设计

**注册机制：** 算法通过统一的注册表加入对比平台。每个算法提供一个运行函数，签名统一为 `(Maze) → AlgorithmResult`。新增算法只需一行注册调用，即可出现在对比面板的勾选列表中。

**隔离运行：** 每个算法拿到 Maze 对象的深拷贝，互不干扰。在后台线程中顺序执行，通过回调更新 UI 进度条。

**结果展示：** QTableWidget 展示所有指标 × 所有算法的矩阵。每个指标的最优值绿色加粗标注。底部自动生成分析文本，如"贪心效率是A*的4.5倍，但步数多88步，适合资源优先场景"。

**图表导出：** 一键将对比结果导出为 matplotlib 的 2×3 子图柱状图，保存 PNG，直接嵌入课程报告。

**路径叠加图：** 算法对比平台增加"路径叠加"模式——将两个算法的轨迹用不同颜色绘制在同一张迷宫地图上，直观展示贪心绕路收集资源 vs A*/BFS 直走最短路径的差异。用两张并排图对比，或用两种颜色叠加在同一张图上。

### 7.4 在报告中的对应关系

| 任务书报告要求 | 平台产出 |
|--------------|----------|
| 以图或表的形式展示实验数据 | 对比结果表格截图 + 导出柱状图 |
| 算法对比分析 | 自动分析文本 → 润色后写入报告 |
| 创新性分析 | "贪心在资源优先场景下效率是A*的X倍"等定量结论 |
| 复杂度分析 | 对比表中执行时间列 + 各算法理论复杂度 |

## 八、创新方向

以下方向均**不改变对外接口**，在现有架构内增强。

### 8.1 自适应贪心权重

基础贪心的性价比公式是静态的。引入**上下文感知的权重函数**：公式中的金币权重和陷阱权重根据当前资源量、距离终点远近、剩余BOSS数量动态调整。资源紧张时激进捡金币，接近终点时保守冲线。训练迷宫集上用网格搜索自动调优权重查找表。报告中作为"自适应 vs 静态"的消融实验呈现。

### 8.2 增量式地图记忆

AI视野仅3×3，但走过的格子不应"忘记"。维护一张已知世界地图，初始全为未知，随探索逐步填充。决策时同时查询两层信息：视野层（实时感知新资源）+ 记忆层（回溯已知但未捡的资源）。本质上将"在线贪心"升级为"渐进式全局贪心"。

### 8.3 有限视野内的多步前瞻

单步贪心可能被"墙后的金币"欺骗——视野中看到一个相邻金币，但实际需要绕远路。在每个决策点，对视野内的每个候选资源用BFS计算真实可达路径长度，替代曼哈顿距离。计算量极小（3×3视野内BFS是O(9)），但彻底消除穿墙误判。

### 8.4 对抗性迷宫的鲁棒策略

交叉测试中，迷宫设计方可能采用对抗性布局（死胡同金币、陷阱-金币夹心、远距诱饵、BOSS堵路等）。AI集成了针对这些模式的检测和应对规则：计算绕路代价、净收益判断、时间折扣因子、BOSS前谨慎策略等。

### 8.5 多目标帕累托分析视角

资源收集最大化和步数最小化是两个冲突目标。将它们显式建模为帕累托前沿问题，不追求单一最优解，而是报告AI在帕累托前沿上的位置，与动态规划全局最优的差距。作为报告中"理论分析"部分的素材。

### 8.6 迷宫难度评估器

在运行探险前，根据迷宫静态属性给出难度评分。评估维度包括：死胡同率（金币放在死胡同深处的比例）、陷阱密度与夹心率（陷阱紧邻金币的次数）、BOSS位置阻塞程度（BOSS是否在必经之路上）、金币密度、最短路径长度与实际资源分布的偏离程度。难度评分可以解释"为什么在这个迷宫上分数低"，避免单纯归咎于算法。

### 8.7 参数消融实验框架

系统化地自动对比不同参数配置下（视野半径 1/2/3、性价比公式变体、权重调整开关）同一算法在同一迷宫集上的表现。自动生成消融实验表格（每一行是一个参数变体，每一列是一个迷宫，单元格是效率比值），直接填入报告的实验分析章节。本质是一个嵌套循环——外层参数组合，内层迷宫集，自动运行并汇总。

### 8.8 AI停留热力图

在 Matplotlib 输出中增加一种图表类型——将 AI 在迷宫中的路径转化为热力图：频繁经过的格子（如往返路口、资源富集区）用暖色，从未经过的格子用冷色，墙壁用黑色。热力图直观展示 AI 的"探索偏好"，可用于分析贪心策略是否在特定区域无效徘徊。

### 8.9 迷宫特征-分数关联分析

在用多个迷宫（来自不同迷宫设计组）完成测试后，做散点图或回归分析：X 轴 = 迷宫某特征（如陷阱密度、BOSS数量、最短路径长度），Y 轴 = AI 效率比值。每个点代表一个迷宫，可以按迷宫设计组着色。定量解释"为什么我们的算法在 A 类迷宫上表现好、在 B 类迷宫上表现差"。

### 8.10 策略可视化回放对比（与 7.3 路径叠加图联动）

在 PyQt 算法对比标签页中，选中两个算法的对比结果后，点击"路径叠加"按钮——两个算法的轨迹用不同颜色绘制在同一张迷宫图上，可逐帧同步回放，直观展示贪心绕路收资源 vs A* 直走最短路径的行为差异。

## 九、验收保障设计

### 9.1 阶段测试保障 — API契约

在 Maze 模型层内置校验器，在数据加载后、算法运行前自动校验格式和连通性。`tests/` 下准备与阶段测试用例同构的自测试用例，确保输入输出协议不出错。

### 9.2 交叉测试保障 — 分层防御

三层策略逐级降级：
- **Layer 1（默认）**：自适应贪心 + 记忆 + A* → 追求高分
- **Layer 2（触发）**：连续N步资源无增长 → 纯A*直通，路径上顺手捡，不绕路 → 保证有分数
- **Layer 3（兜底）**：BOSS战失败且金币不足，或遇必经BOSS且资源不够复活 → BFS路径直行，遇BOSS仅用无冷却普攻；复活失败则产出明确"不可通过"结果 → 保证不进入死循环，也不会假死

每一层都是完整可独立工作的策略。Layer 3 的关键是"永不卡死"——即使 BOSS 堵在唯一通路上，系统也有确定的终止路径（战斗到最后一刻并产出结果），而非无限循环或挂起。

### 9.3 自动化回归测试框架

每次修改代码后，一键运行全部测试用例（空迷宫、密集金币、密集陷阱、单/多BOSS、长路径、多分支等）。自动校验输出格式、路径合法性、分数不低于基线。发现问题立即报警。

### 9.4 交叉测试预检工具

在正式交叉测试前，自动生成5类典型对抗性迷宫（死胡同金币、陷阱夹心、BOSS堵路、长绕路、分散陷阱），运行AI并输出分数预估。分数不够时提前调整策略参数。

**9.7.2 自动验证流程**

每次验证时：
1. 加载 `cases/` 下所有 JSON 用例
2. 逐个运行对应的算法模块
3. 与 `expected/` 中的预期结果比对（允许浮点误差）
4. 输出通过/失败列表，失败的标出差异

验证可作为独立脚本运行：`python -m tests.self_validation.run`，不依赖 PyQt，仅依赖算法核心层。

**9.7.3 BOSS 战正确性验证的特殊处理**

分支限界的正确性验证方式：
- **小规模用例（BOSS血量 ≤ 30，技能 ≤ 2）**：同时运行暴力枚举（穷举所有技能序列至最大回合数），对比分支限界结果与暴力枚举结果完全一致。
- **大规模用例（BOSS血量 > 30）**：暴力枚举不可行，改用"技能序列模拟执行验证"——将分支限界输出的技能序列逐回合模拟，确认最终BOSS血量 ≤ 0 且回合数 ≤ 限定回合数，且无冷却违规。同时与迷宫设计方提供的预期最少回合数比对。

**9.7.4 贪心拾取正确性验证**

手工构造确定性的 3×3 局部迷宫，精确计算每个候选资源的性价比：
- 用例 `greedy_simple.json`：视野内仅 1 个金币，结果唯一
- 用例 `greedy_multi.json`：视野内 2 金币（距离不同）+ 1 陷阱，验证排序顺序和排除逻辑
- 用例 `greedy_tiebreak.json`：两个等性价比金币，验证确定性选择（优先近终点方向）

**BOSS战可考据性保障：** 分支限界的正确性通过（9.7.3自主验证 + 与暴力枚举对比）保证。报告中可写出伪代码——从状态定义到剪枝条件——逻辑完整、可复现。

## 十二、关键设计决策汇总

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| 1 | 视野范围 | 默认 3×3（r=1） | 与任务书局部迷宫 3×3 一致，且参数可调 |
| 2 | 性价比公式 | 价值 / (距离 + 1) | 简洁、可解释、符合贪心定义 |
| 3 | 陷阱性价比 | 负数直接排除 | 贪心选正收益目标，不应主动走向陷阱 |
| 4 | BOSS战 | 分支限界求最优 + 失败可复活 | 任务书明确要求，复活机制增加容错 |
| 5 | 视野外记忆 | 维护已知世界地图 | 用信息增量弥补视野限制 |
| 6 | 降级策略 | 三层防御 | 确保交叉测试任何迷宫上不会零分 |
| 7 | 可视化架构 | Console + PyQt + Matplotlib 三层 | 覆盖开发→答辩→报告全链路 |
| 8 | 外部格式对接 | Parser 策略模式 + 符号→整数转换层 | 外部 JSON 使用人类可读的字符符号，Parser 层统一转为内部整数编码，格式变化仅改 Parser |
| 9 | 算法对比 | 注册机制 + deepcopy隔离 | 新增算法一行注册，运行互不干扰 |
| 10 | PyQt架构 | QThread + 信号/槽 | 算法实时运行 + UI不卡死 + 可暂停调速 |
| 11 | 探险循环编排 | QThread + Strategy 接口注入，不硬编码算法 | 新增策略 = 写一个Strategy子类；Worker和exploration零改动 |
| 12 | 贪心返回值 | 扩展为决策详情（候选列表+性价比+选中目标） | 前端展示推理过程，核心算法逻辑不变 |
| 13 | A*缓存策略 | 不缓存，每步重算 | 15×15上<1ms，且迷宫状态可能变化 |
| 14 | BOSS战架构 | 求解模式与执行模式分离 | 对比平台走求解，实时展示走执行+回调 |
| 15 | 图片渲染 | 双层方案：几何图形为底，图片为可选增强 | 图片缺失自动回退，不增加硬依赖 |
| 16 | BOSS技能数 | 冷却列表动态长度，匹配 `PlayerSkills` 数组长度 | 适应不同迷宫设计组的技能配置 |
| 17 | BOSS终止响应 | progress_callback 返回布尔值控制继续/终止 | 避免BOSS战中用户请求终止需等待战斗结束 |
| 18 | BOSS前置检查 | 遇BOSS先判断资源是否够复活，不够则提前降级 | 避免先打再失败，浪费执行 |
| 19 | Layer 3定义 | BFS+普攻+复活耗尽后产出明确结果 | 保证任何迷宫上都可终止，不假死不循环 |
| 20 | BOSS状态表示 | 从单字符串改为结构列表（支持多BOSS并存） | 一个迷宫可能2~3个BOSS，各自跟踪独立状态 |
